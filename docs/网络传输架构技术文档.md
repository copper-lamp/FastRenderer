# 网络传输架构 — 技术验证总结

## 概述

本阶段完成了**完全网络传输**架构的技术验证：所有 GUI 定义、按键注册、计算任务均通过 Minecraft 自定义网络数据包（Custom Packet）在 JS 插件和 C++ 渲染引擎之间传输，消除了硬编码的 C++ 端 UI 定义。

### 数据流模式

JS Plugin 通过文件桥接（`.calc_bridge/` 目录）与 C++ 通信，C++ 桥接线程将文件内容转为 Minecraft 自定义 Packet 广播到所有客户端：

| 文件 | → | Packet | 方向 | 用途 |
|---|---|---|---|---|
| `keybind_pending.json` | → | `RegisterKeybindPacket` | Server→Client | 按键注册 |
| `gui_pending.json` | → | `GuiDefinitionPacket` | Server→Client | GUI 定义分发 |
| `request.json` | ← | `CalcRequestPacket` | Client→Server | 计算请求 |
| `response.json` | → | `CalcResponsePacket` | Server→Client | 计算结果 |

## 二、核心组件

### 2.1 自定义数据包 (Custom Packets)

定义在 [CalcPackets.h](file:///d:/FastRenderer/ChiyanMap/src/calc/CalcPackets.h)，继承 `ll::network::Packet`：

| Packet | 字段 | 序列化 | 处理器 |
|---|---|---|---|
| `CalcRequestPacket` | requestId, a, b, op | writeVarInt, writeFloat×2, writeString | `CalcRequestHandler` |
| `CalcResponsePacket` | requestId, result, error | writeVarInt, writeFloat, writeString | `CalcResponseHandler` |
| `GuiDefinitionPacket` | guiId, jsonContent | writeString×2 | `GuiDefinitionHandler` |
| `RegisterKeybindPacket` | pluginId, bindId, bindName, vk | writeString×3, writeVarInt | `RegisterKeybindHandler` |

自动注册方式：
```cpp
inline bool g_calcPacketsRegistered = []() {
    auto& reg = ll::network::PacketRegistrar::getInstance();
    uint64 hash = std::hash<std::string_view>{}("PacketName");
    reg.registerPacket("PacketName", hash,
        []() -> std::unique_ptr<ll::network::Packet> { return std::make_unique<PacketClass>(); },
        &handlerInstance);
    return true;
}();
```

### 2.2 文件桥接 (CalcBridge)

定义在 [CalcPackets.h:CalcBridge](file:///d:/FastRenderer/ChiyanMap/src/calc/CalcPackets.h#L162-L311) 命名空间：

- **目录**: `Mods/ChiyanMap/.calc_bridge/`
- **初始化**: 自动创建目录 + 清空残留文件
- **桥接线程**: 独立线程 `bridgeLoop()`，每 200ms 轮询：
  - `pollGuiDefinitions()` — 读取 `gui_pending.json` → 保存文件 + 广播 `GuiDefinitionPacket` + 内存注入
  - `pollKeybindRegistrations()` — 读取 `keybind_pending.json` → 广播 `RegisterKeybindPacket`
  - `pollCalcResponses()` — 读取 `response.json` → 直接调用 `g_calcResponseCallback` 更新前端

### 2.3 JSON GUI 系统

三层架构：

| 层 | 文件 | 职责 |
|---|---|---|
| **Parser** | [JsonGuiParser.h](file:///d:/FastRenderer/ChiyanMap/src/renderer/JsonGuiParser.h) | JSON → `GuiDefinition` 结构体（支持文件和字符串两种来源） |
| **Renderer** | [JsonGuiRenderer.h](file:///d:/FastRenderer/ChiyanMap/src/renderer/JsonGuiRenderer.h) | `GuiDefinition` → ImGui 控件渲染，包含 DynamicText、事件回调 |
| **Loader** | [JsonGuiLoader.h](file:///d:/FastRenderer/ChiyanMap/src/renderer/JsonGuiLoader.h) | 文件监控、热加载、内存注入、生命周期管理 |

关键特性：
- **文件热加载**: 监控 `gui_defs/` 目录，`.json` 文件变更时自动重载
- **内存注入**: `injectDefinition(guiId, jsonContent)` 不经过文件系统
- **DynamicText**: JSON GUI 中 `ref` 字段绑定 `DynamicText::values["ref_name"]`，C++ 可实时更新文本
- **事件回调**: `setEventCallback()` 将 JSON 控件事件映射到 C++ 处理函数

### 2.4 按键注册系统

定义在 [MapRenderState.h:KeybindingRegistry](file:///d:/FastRenderer/ChiyanMap/src/state/MapRenderState.h#L14-L77)：

- `registerKeybind(pluginId, id, name, vk, callback)` — 注册按键
- `isRegistered(vk)` / `isIdRegistered(id)` — 检查是否已注册
- 按键检测由 `RenderVerificationFrame()` 中的直接 `GetAsyncKeyState()` 边缘检测处理（避免 inline 变量跨 TU 问题）

### 2.5 DX11 键盘钩子

定义在 [DX11Hook.h](file:///d:/FastRenderer/ChiyanMap/src/hooks/DX11Hook.h)：

- `hkGetAsyncKeyState`: 拦截系统 API，UI 激活时除白名单键（K/N/M/U/F11）外的按键均返回 0
- `hkGetKeyState`: 同上
- 白名单键（K/N/M/U）由 `RenderVerificationFrame` 的直接边缘检测处理，不依赖 `KeybindingRegistry`

### 3.2 验证中遇到的问题

| 问题 | 原因 | 修复 |
|---|---|---|
| JS 插件无法加载 | manifest.json type 为 "native" | 改为 `"type": "lse-nodejs"` |
| `file` 未定义 | LegacyScriptEngine Node.js 模式无 `file` API | 改用 `require('fs')` |
| 所有 GUI 无法打开 | inline 变量跨 TU 重复副本（MSVC DLL） | 用直接 `GetAsyncKeyState` 边缘检测代替 `pollAll` |
| K 键无法关闭 GUI | `hkGetAsyncKeyState` 阻塞未注册的键 | 白名单 K/N/M/U |
| 计算器 GUI 不显示 | 文件监控线程擦除了内存注入的定义 | 注入时同时保存文件到 `gui_defs/`，`scanDirectory` 保留 `"n:"` 定义 |
| 计算结果超时 | `CalcRequestHandler` 阻塞网络线程 | 改为 bridgeLoop 异步轮询 `response.json` |
| 重复控件 ID | 文件定义 + 注入定义重复渲染 | `scanDirectory` 按 `id` 去重 |
| 鼠标双指针 | `while(ShowCursor)` 循环暴力隐藏/显示 | 改为单次调用 |
| 计算器定义双重渲染 | `scanDirectory` 保留了语义重复定义 | 保留时检查 `nd.id == oldDef.id` |

### 3.3 关键教训：MSVC DLL Inline 变量

**这是最严重的问题。** 在 MSVC DLL 构建中，头文件中定义的 `inline` 变量可能在不同翻译单元（Translation Unit）间产生独立副本，导致状态不一致。

受影响的范围：
- `KeybindingRegistry::g_keybinds` — 按键注册表（修复：直接 GetAsyncKeyState 检测）
- `JsonGuiLoader::g_definitions` — GUI 定义池（修复：Meyer's Singleton 模式）

**解决方案**: 使用 Meyer's Singleton（静态局部变量 + inline 访问函数）替代 inline 变量：

```cpp
// ❌ 不要用 inline 变量共享跨 TU 状态
inline std::vector<GuiDefinition> g_definitions;

// ✅ 使用 Meyer's Singleton
inline State& getState() {
    static State s;
    return s;
}
// 后续通过 getState().definitions 访问
```

C++ 标准保证 `static` 局部变量在 inline 函数中跨 DLL 边界只有一个实例。

## 四、JS 插件开发指南

### 4.1 插件注册

```javascript
// manifest.json — LegacyScriptEngine Node.js 模式
{
    "entry": "index.js",
    "name": "plugin_name",
    "type": "lse-nodejs",
    "version": "1.0.0"
}

// index.js — 必须调用 registerPlugin
ll.registerPlugin("plugin_name", "description", [1,0,0], {});
```

### 4.2 文件桥接 API

桥接目录 = `{ServerRoot}/Mods/ChiyanMap/.calc_bridge/`

| 文件 | 格式 | 说明 |
|---|---|---|
| `keybind_pending.json` | `{"pluginId","bindId","bindName","vk"}` | C++ 轮询后自动删除 |
| `gui_pending.json` | `{"id":"xxx","root":{...}}` | 完整 GUI 定义 JSON |
| `request.json` | `{"id","op","a","b"}` | 由 C++ 写入，JS 读取 |
| `response.json` | `{"id","result","error"}` | 由 JS 写入，C++ 读取 |

### 4.3 计算插件示例

```javascript
const fs = require('fs');
const path = require('path');

// 推导桥接目录路径
const PLUGIN_DIR = __dirname.replace(/\\/g, "/");
const SERVER_ROOT = path.resolve(PLUGIN_DIR, "../../").replace(/\\/g, "/");
const BRIDGE = SERVER_ROOT + "/Mods/ChiyanMap/.calc_bridge";

// 轮询计算请求
setInterval(() => {
    const reqPath = BRIDGE + "/request.json";
    if (fs.existsSync(reqPath)) {
        const req = JSON.parse(fs.readFileSync(reqPath, 'utf-8'));
        fs.unlinkSync(reqPath);
        const result = compute(req.op, req.a, req.b);
        fs.writeFileSync(BRIDGE + "/response.json", JSON.stringify({
            id: req.id, result: result, error: ""
        }));
    }
}, 100);
```

## 五、开发注意事项

### 5.1 路径约定

| 路径 | 用途 |
|---|---|
| `Mods/ChiyanMap/.calc_bridge/` | 文件桥接目录（C++ + JS 共享） |
| `Mods/ChiyanMap/gui_defs/` | GUI 定义持久化目录（文件监控） |

路径推导规则：
- **JS 端**: `__dirname` → 上两级得到 Server Root
- **C++ 端**: `mSelf.getModDir()` → 拼接 `.calc_bridge`/`gui_defs`

### 5.2 线程模型

| 线程 | 运行内容 |
|---|---|
| **主线程** | DX11 Hook → ImGui 渲染 |
| **网络线程** | 自定义 Packet 收发（不应阻塞） |
| **桥接线程** | 每 200ms 轮询桥接目录 |
| **文件监控线程** | 每 500ms 扫描 `gui_defs/` |
| **JS 事件循环** | JS 插件的 setInterval/setTimeout |

### 5.3 已知限制

1. `sendToServer()` / `sendToClient()` 在极早期可能未初始化
2. 计算请求的 `response.json` 轮询在 bridgeLoop（独立线程），响应不经过网络
3. `CalcRequestHandler` 已改为非阻塞模式——只写 `request.json` 立即返回

## 六、进入正式开发的建议

### 6.1 架构完善优先级

1. **Plugin SDK 封装** — 将文件桥接、Packet 注册抽象为 JS 可调用的 API
2. **插件管理器** — 统一的插件注册、生命周期、依赖管理
3. **GUI 组件库** — 预置常见的 JSON GUI 组件模板
4. **错误恢复机制** — 桥接目录损坏恢复、Packet 重试、超时处理
5. **多客户端支持** — `sendToClient` 针对特定 NetworkIdentifier 路由

### 6.2 建议的工程规范

- 所有头文件中的共享状态必须使用 Meyer's Singleton
- JS 插件统一使用 `require('fs')` 而非 LegacyScriptEngine 专有 API
- 自定义 Packet 的 hash 字符串必须和注册时完全一致
- 不要在网络线程中执行阻塞操作
